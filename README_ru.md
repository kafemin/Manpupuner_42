## 📖 О проекте

**Manpupuner_42** — это демонстрация концепции (Proof of Concept), которая бросает вызов устоявшимся подходам в разработке операционных систем. Вместо эмуляции или виртуализации, проект предлагает **унификацию интерфейса** на самом низком уровне — уровне системных вызовов.

**Суть идеи:**  
Создать минимальную прослойку (арбитр) между аппаратным обеспечением и приложениями. Этот арбитр понимает запросы из разных экосистем (например, POSIX и NT) и обрабатывает их через **единый внутренний API из 7 базовых вызовов**.

> 🚀 **Это не эмуляция, не виртуализация и не второе ядро.**  
> Это — **единое ядро с двумя интерфейсами**, обеспечивающее двунаправленную совместимость.

## 🧱 Архитектура: Три уровня унификации

Система строит "мост" между двумя разными мирами через три логических уровня:

```text
┌─────────────────────────────────────────────────────────────┐
│                   ПРИЛОЖЕНИЯ / APPLICATIONS                  │
├──────────────────────────┬──────────────────────────────────┤
│  Слой A (POSIX-стиль)    │  Слой B (NT-стиль)              │
│  open(), read(), fork()  │  CreateFile(), ReadFile(), ...  │
├──────────────────────────┴──────────────────────────────────┤
│               ЕДИНЫЙ АРБИТР / UNIFIED ARBITER               │
│         7 базовых вызовов / 7 basic calls:                  │
│   READ_FILE, WRITE_FILE, CREATE_PROCESS, SLEEP,            │
│   ALLOC_MEMORY, FREE_MEMORY, LIST_FILES                    │
├─────────────────────────────────────────────────────────────┤
│               ИСПОЛНИТЕЛЬ (ЭМУЛЯЦИЯ) / EXECUTOR            │
│         Виртуальная ФС, память, процессы, время            │
└─────────────────────────────────────────────────────────────┘
```

## 🔧 Внутренний API: 7 Столпов Системы

Внутренний API арбитра — это "ядро" проекта. Он содержит всего 7 вызовов, которые покрывают все базовые потребности любой программы:

| № | Вызов | Описание | Пример использования |
|:-:|:---|:---|:---|
| 1 | `READ_FILE` | Чтение данных из виртуальной ФС | `READ_FILE(path="/home/file.txt")` |
| 2 | `WRITE_FILE` | Запись данных в виртуальную ФС | `WRITE_FILE(path="/home/file.txt", data="Hello")` |
| 3 | `CREATE_PROCESS` | Создание нового процесса (эмуляция) | `CREATE_PROCESS(name="my_app")` |
| 4 | `SLEEP` | Приостановка выполнения | `SLEEP(seconds=2)` |
| 5 | `ALLOC_MEMORY` | Выделение блока виртуальной памяти | `ALLOC_MEMORY(size=1024)` |
| 6 | `FREE_MEMORY` | Освобождение блока памяти | `FREE_MEMORY(id=0)` |
| 7 | `LIST_FILES` | Список файлов в виртуальной ФС | `LIST_FILES()` |

## 💻 Пример использования: Как это работает

Этот пример показывает, как два разных интерфейса (POSIX и NT) приводятся к одному и тому же внутреннему действию.

```python
from hybrid_arbiter_ru import HybridArbiter, PosixCompatLayer, NtCompatLayer

# 1. Создаём ядро-арбитр
kernel = HybridArbiter()

# 2. Создаём два слоя совместимости
posix = PosixCompatLayer(kernel)
nt = NtCompatLayer(kernel)

# 3. POSIX-программа (Linux-стиль)
fd = posix.open("/home/user/hello.txt")  # Возвращает файловый дескриптор
content = posix.read(fd, 50)             # Читает файл
posix.close(fd)                          # Закрывает файл

# 4. NT-программа (Windows-стиль)
handle = nt.CreateFile("C:\\Users\\User\\hello.txt")  # Возвращает HANDLE
content = nt.ReadFile(handle, 50)                    # Читает файл
nt.CloseHandle(handle)                               # Закрывает файл
```

**В чём магия?** Внутри арбитра оба вызова (`open()` и `CreateFile()`) транслируются в один и тот же внутренний вызов `READ_FILE`. Арбитр не знает, откуда пришёл запрос — он просто выполняет команду.

## 📊 Сравнение с существующими подходами

| Подход | Принцип | Недостаток / Отличие от Manpupuner_42 |
|:---|:---|:---|
| **Гибридное ядро** (Windows NT, macOS XNU) | Одно ядро, одна экосистема | Не умеет запускать приложения из других систем |
| **Односторонняя трансляция** (WSL 1) | Транслирует вызовы A → B | Работает только в одну сторону |
| **Виртуализация** (WSL 2, VirtualBox) | Запускает полноценное ядро гостевой ОС | Два ядра, высокие накладные расходы |
| **Эмуляция API** (Wine) | Эмулирует API в пользовательском режиме | Работает на уровне API, а не ядра |
| **Manpupuner_42** | **Одно ядро — два интерфейса** | ✅ **Двунаправленная совместимость на уровне ядра** |
