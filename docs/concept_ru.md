## 1. Введение: проблема и идея

Современные операционные системы строятся вокруг **ядра** — программы, которая управляет аппаратным обеспечением и предоставляет приложениям **системные вызовы** (интерфейс для работы с файлами, памятью, процессами и т.д.).

Проблема в том, что разные ОС используют **разные наборы системных вызовов**. Программа, написанная для одной системы, не может работать на другой без эмуляции или переписывания кода. Это создаёт барьеры для разработчиков, пользователей и экосистем в целом.

**Главная идея проекта Manpupuner_42:**

> Создать **единую минимальную прослойку** (арбитр) между аппаратным обеспечением и системными вызовами, которая понимает запросы из разных экосистем и обрабатывает их через общий внутренний API.

Это не эмуляция, не виртуализация и не второе ядро. Это — **унификация интерфейса** на самом низком уровне.

---

## 2. Название проекта: Manpupuner_42

Название **Manpupuner_42** вдохновлено двумя символами:

1. **Маньпупунёр** — геологический памятник на Северном Урале. Семь каменных столбов-великанов высотой до 42 метров стоят как незыблемые стражи, символизируя фундаментальные принципы, на которых строится новая архитектура.
2. **42** — универсальное число, отсылающее к поиску ответа на главный вопрос. В нашем случае — это поиск универсального решения для совместимости систем.

> **Символизм:** Как семь столбов Маньпупунёра стоят на труднодоступном плато, так и семь базовых системных вызовов становятся фундаментом новой архитектуры — простой, понятной и универсальной.

## 3. Архитектура системы: три уровня

Система Manpupuner_42 состоит из трёх логических уровней:

### Уровень 1: Внешние интерфейсы (API-слои)

Это слои, которые принимают запросы от приложений в формате конкретных ОС. Каждый слой **транслирует** внешний вызов во внутреннее представление арбитра.

- **Слой POSIX:** Понимает вызовы типа `open()`, `read()`, `fork()`, `exec()`, `sleep()`.
- **Слой NT:** Понимает вызовы типа `CreateFile()`, `ReadFile()`, `CreateProcess()`, `Sleep()`.

Эти слои не являются ядрами. Это небольшие библиотеки-трансляторы, которые преобразуют синтаксис и параметры вызовов в единый формат.

### Уровень 2: Единый арбитр (ядро)

Это сердце системы. Арбитр принимает **внутренние запросы** (унифицированные) и выполняет их через собственный минимальный набор функций.

Внутренний API арбитра содержит 7 базовых операций:

| № | Внутренний вызов | Что делает |
|:-:|:---|:---|
| 1 | `READ_FILE` | Читает данные из виртуальной файловой системы |
| 2 | `WRITE_FILE` | Записывает данные в виртуальную ФС |
| 3 | `CREATE_PROCESS` | Создаёт новый процесс (эмуляция) |
| 4 | `SLEEP` | Приостанавливает выполнение на заданное время |
| 5 | `ALLOC_MEMORY` | Выделяет блок виртуальной памяти |
| 6 | `FREE_MEMORY` | Освобождает блок памяти |
| 7 | `LIST_FILES` | Возвращает список файлов в виртуальной ФС |

Арбитр не знает, откуда пришёл запрос — из POSIX или NT. Он просто выполняет команду.

### Уровень 3: Исполнитель (эмуляция железа)

На этом уровне арбитр взаимодействует с «железом». В текущей демонстрации железо **эмулируется**:

- Вместо реальной файловой системы — словарь в памяти.
- Вместо реальных процессов — счётчик и список имён.
- Вместо реального времени — `time.sleep()`.

Это сделано намеренно, чтобы показать **принцип**, а не создавать готовую ОС.

## 4. Принцип работы: пошагово

Рассмотрим, как система обрабатывает один запрос:

1. **Приложение** вызывает функцию (например, `open("/home/file.txt")` в стиле POSIX или `CreateFile("C:\\file.txt")` в стиле NT).
2. **Слой совместимости** перехватывает вызов, извлекает параметры и формирует **внутренний запрос** к арбитру: `{ type: READ_FILE, args: { path: "/home/file.txt" } }`.
3. **Арбитр** получает запрос, определяет его тип и вызывает соответствующий обработчик.
4. **Исполнитель** выполняет операцию (например, ищет файл в виртуальной ФС и возвращает содержимое).
5. **Арбитр** возвращает результат обратно в слой совместимости.
6. **Слой совместимости** преобразует результат в формат, ожидаемый приложением (например, файловый дескриптор или HANDLE).
7. **Приложение** получает результат и продолжает работу.

**Ключевой момент:** Для арбитра неважно, откуда пришёл запрос. Вызов `open()` из POSIX и `CreateFile()` из NT приводятся к одному внутреннему вызову `READ_FILE`. Это и есть **единство системных вызовов**.

## 5. Что доказывает демонстрация

Демонстрационная реализация на Python показывает:

1. ✅ **Распознаваемость:** Арбитр понимает команды из двух разных «миров» (POSIX и NT).
2. ✅ **Единообразие:** Разные внешние вызовы сводятся к одному внутреннему API.
3. ✅ **Работоспособность:** Система выдаёт ожидаемый результат (содержимое файла, создание процесса, пауза).
4. ✅ **Минимализм:** Для работы достаточно 7 базовых вызовов. Всё остальное может быть реализовано поверх них.

Это доказательство того, что **принцип универсального арбитра работает**.

## 6. Отличие от существующих подходов

| Подход | Принцип | Отличие от Manpupuner_42 |
|:---|:---|:---|
| **Гибридное ядро** | Одно ядро, одна экосистема | Не умеет запускать приложения из других систем |
| **Трансляция (односторонняя)** | A → B (только в одну сторону) | Только одно направление |
| **Виртуализация** | Два ядра, гипервизор | Требует виртуализации |
| **Эмуляция API** | Пользовательский режим | Работает на уровне API, а не ядра |
| **Гостевое ядро** | Два ядра, одно — «гость» | Нет единого арбитра |
| **Manpupuner_42** | **Единое ядро-арбитр** с двумя слоями трансляции | **Двунаправленная совместимость на уровне ядра** |

## 7. История создания

### 🧠 Исходная концепция (автор: Каськов Александр)

Идея проекта родилась из наблюдения:

> *«Если программы обращаются к ядру через системные вызовы, почему бы не создать единую прослойку, которая понимает вызовы из разных систем и выполняет их через общий внутренний API?»*

Автор предположил, что:

- Два ядра не нужны — достаточно одного арбитра.
- Конфликты между ядрами можно обойти, если убрать сами ядра и оставить только системные вызовы.
- Такой подход сократит объём кода и упростит разработку кроссплатформенного ПО.

**Двухэтапная проверка концепции:**

1. **Python PoC** — концепция проверена на Python: [https://github.com/kafemin/Manpupuner_42](https://github.com/kafemin/Manpupuner_42)
2. **Реализация ядра** — на C и ассемблере: [https://github.com/kafemin/manpupuner_kernel](https://github.com/kafemin/manpupuner_kernel)

### 💻 Техническая реализация

Автор обратился к AI-ассистенту **DeepSeek** с запросом на создание работающей демонстрации. В ходе диалога были сформулированы и реализованы:

1. Архитектура трёх уровней
2. Внутренний API из 7 базовых вызовов
3. Два слоя совместимости (POSIX и NT)
4. Виртуальная файловая система в памяти
5. Набор тестовых сценариев
